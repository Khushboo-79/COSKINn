# COSKINn — Admin Panel Build Plan (React + Tailwind)

**Repo path:** `apps/internal-panel/src/modules/admin`
**Stack:** React + Tailwind CSS · shared `packages/ui` component library · React Query · React Hook Form + Zod · shared route shell (login, nav, layout) inside `apps/internal-panel`
**Depends on backend:** `auth/RBAC` + 2FA (`02-BACKEND-BUILD-PLAN.md` Day 9), plus **read access into every other module** (product Days 5–9, inventory Day 31, order Day 27, finance Days 53–56, HR Days 57–58, marketing Day 62, support Days 60–61, audit Day 59) and **write access into** `roles/permissions/user_roles/role_permissions` (Day 9), product approval gate (Day 5–6), wallet expiry policy (Day 43), membership tier thresholds (Day 50)
**Connects to panels** (per `00-PANEL-CONNECTIVITY-MAP.md` §2):
- **Sends to →** none as a data producer — top-level oversight; its only "output" to another panel is the category/brand and product-approval decision Product Management reads back, and the RBAC/role writes every panel's guard depends on
- **Receives from ←** **all panels** — Product Management (approval requests), Inventory (low-stock/near-expiry feed), Order Management (order/sales rollup), Tax/Finance (P&L, GST dashboards), HR (headcount, payroll cost), Marketing/CRM (campaign/channel performance), Customer Support (SLA/ticket volume), Auditor (available for cross-reference, though Admin has its own read access rather than depending on Auditor)

This is the **only panel with full read access across every module and write access to role/permission data** — every other panel's RBAC guard ultimately traces back to a role this panel manages. All cross-panel reads happen only through backend aggregation APIs — this panel never calls another panel directly.

Per the build-order dependency chain (`00-PANEL-CONNECTIVITY-MAP.md` §4), Admin is **built in parallel throughout**, not as a single phase — its scaffold, RBAC/role management, and approval-gate screens (Days 1–5 below) must exist early since Product Management's Day 9 approval workflow depends on them, but its dashboard widgets (Days 6–12) go live incrementally as each source panel/module ships real data. Treat Days 1–5 as Phase-1/2 priority work and Days 6–16 as work that trails each dependency's real ship date.

**Compliance/data-integrity note:** this module is one of two internal panels (with Tax/Finance) requiring mandatory 2FA per the non-functional targets in `02-BACKEND-BUILD-PLAN.md` — the login flow must not allow bypass, since this is also the one panel that can write to `role_permissions`/`user_roles`. Every write this panel makes to another module's data (category/brand approval, product-Live approval, wallet expiry policy, membership thresholds) must go through that module's own backend endpoint, never a direct table write, and must be captured in `audit_logs` (Backend Day 59) so the Auditor Panel's admin-activity-log screen has a complete record. Dashboard widgets are read/aggregate-only — this panel must never let a widget double as an editable form for another module's data.

**Definition of done:** role & permission management works end-to-end with changes correctly propagating to every panel's RBAC guard, category/brand and product-Live approval gates are enforced server-side and unblock Product Management with no manual database step, every dashboard widget in this plan reflects live (non-mocked) data from its source module by that module's ship date, wallet-expiry and membership-tier config are writable only from this panel, and the admin-activity audit trail (feeding Auditor Panel) captures every write this panel makes.
